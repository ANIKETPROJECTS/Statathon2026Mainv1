### The Pattern — This Is the Logic

**Rows 1–12 are FRAME/DESIGN variables.** They identify the sampling unit (FSU, State, District etc.). These come from the survey's sampling frame, not from any section of the questionnaire. So Sec, Item, Col are **blank** for them.

**Rows 13–16 and 19–20 are QUESTIONNAIRE variables.** They are actual answers recorded by the enumerator on Section 1 of the paper questionnaire (HCQ). So they have `Sec = 1` and an `Item` number referencing a specific question.

**Rows 17, 18, 21 are SYSTEM-GENERATED variables.** Questionnaire No., Level, and Multiplier are auto-generated by the data processing system, not entered by the enumerator. So Sec, Item, Col are **blank** for them too.

---

### The Exact Rule to Detect This

The distinguishing signal is **what kind of variable it is**, and you can detect this from the CSV itself using these rules:

**Rule 1 — Single fixed value across ALL rows → System-generated → Sec/Item/Col BLANK**

Check: does this column have exactly 1 unique value (ignoring nulls)?
- `Survey_Name` → always "HCES" → Generated → blank
- `Year` → always "2024" → Generated → blank
- `Questionnaire_No` → always "H" → Generated → blank
- `Level` → always "1" → Generated → blank

**Rule 2 — High null rate (>50% nulls) AND appears in questionnaire section → Sec/Item/Col FILLED**

Check: is this column mostly null but has a questionnaire-linked meaning?
- `Sample_Sub_Division_No` → 99% null → BUT it is a questionnaire field (Section 1, Item 1.10) → Sec=1, Item=1.10 → FILLED
- `Reason_for_Substitution_Code` → 93% null → BUT it is a questionnaire field (Section 1, Item 1.14) → Sec=1, Item=1.14 → FILLED

**Rule 3 — Frame/design variable (geographic/stratification variable) → Sec/Item/Col BLANK**

These are identifiable by their names belonging to the sampling frame vocabulary: FSU, State, District, Stratum, Sub-stratum, Panel, Sub-sample, Region, Sector, FOD. They are never recorded on the questionnaire — they exist in the sampling design frame and are attached to records by the system.

**Rule 4 — Multiplier is always BLANK** regardless of anything else, because it is a system-computed weight, never a questionnaire field.

---

### So the Core Detection Logic Is:

```
For each column, ask:

1. Is this column name in the "frame variable" vocabulary?
   (FSU, State, District, Stratum, Sub-stratum, Panel, Sub-sample, 
    NSS-Region, FOD-Sub-Region, Sector, Survey-Name, Year)
   → Sec = blank, Item = blank, Col = blank

2. Is this column system-generated?
   (Questionnaire No., Level, Multiplier, Year, Survey Name)
   → Sec = blank, Item = blank, Col = blank

3. Does this column have EXACTLY 1 unique value?
   → System-generated → Sec = blank, Item = blank, Col = blank

4. Everything else = it came from the questionnaire
   → Sec = the section number it belongs to
   → Item = the question number (e.g. 1.7, 1.10, 1.11...)
   → Col = the column number within that section's block
```

---

### The Missing Piece: Where Do the Actual Numbers (1.7, 1.10, 1.11...) Come From?

This is the critical point — **these cannot be derived from the CSV alone.** The numbers 1.7, 1.10, 1.11, 1.12, 1.13, 1.14 are question numbers printed on the **physical paper questionnaire**. The CSV has no memory of which question number each column came from.

So you have two options for your software:

**Option A — Manual mapping input.** After auto-generating the table, for any column detected as a "questionnaire variable" (Rule 4), leave Sec/Item/Col highlighted/editable and ask the user to fill them in from the paper questionnaire.

**Option B — Companion mapping file.** Accept a second upload: a small CSV or JSON file that maps `column_name → (Sec, Item, Col)`. For HCES this would be:
```
Sample_SU_No → Sec=1, Item=1.7
Sample_Sub_Division_No → Sec=1, Item=1.10
Second_Stage_Stratum_No → Sec=1, Item=1.11
Sample_Household_No → Sec=1, Item=1.12
Survey_Code → Sec=1, Item=1.13
Reason_for_Substitution_Code → Sec=1, Item=1.14
```

The auto-detection handles Rules 1–3 perfectly with zero human input. Only Rule 4 (questionnaire-linked fields) needs the mapping file or manual entry — because that information lives in the paper form, not in the data.
